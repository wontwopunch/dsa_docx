# DSA 재개발 — 기술 개발 가이드

> 대상: 백엔드/개발팀 내부 참고용
> 기술 스택: Spring Boot + Supabase(PostgreSQL) + Flutter
> 작성 기준: PRD(DSA 재개발 필요성 및 신규 앱 연동 계획) 검토 결과

---

## 1. 문서 목적

PRD에 정의된 기능 요구사항을 실제로 구현하기 전에, 개발팀 관점에서 짚어야 할 선결 조건·아키텍처 원칙·리스크를 정리한다. PRD는 "무엇을 만들 것인가"를 다루고, 이 문서는 "어떻게, 어떤 순서로, 무엇을 먼저 확인하고 만들 것인가"를 다룬다.

---

## 2. 개발 착수 전 선결 조건 (Blocker)

아래 5개 항목은 기술 설계의 입력값이다. 답이 안 나온 상태에서 DB 스키마나 모듈 구조를 확정하면 재작업 위험이 크므로, Phase 0 시작 전에 반드시 확인한다.

| # | 확인 대상 | 확인 내용 | 결정에 따라 바뀌는 것 |
| --- | --- | --- | --- |
| 1 | 디멤버(D멤버) | 개발·소유 주체가 대성전산인지, 연동 방식 A(API유지)/B(직접연동)/C(재개발) 중 무엇인지 | 4.10·4.20·4.21·4.29·4.32 등 7개 이상 메뉴의 개발 범위 |

## 3. 전체 아키텍처 원칙

### 3.1 공통 백엔드 모듈 (Phase 0에서 먼저 구축)

PRD 전반에서 반복되는 패턴 3가지는 메뉴별로 따로 구현하지 말고 공통 서비스로 분리한다.

- **학생 검색 서비스**: 거의 모든 메뉴가 동일한 필터 셋(학과/전형/계열/반/성별/상태 등)을 사용함. `Specification` 또는 QueryDSL 기반 동적 검색 빌더 하나로 통합.
- **엑셀 Import/Export 프레임워크**: 업로드 전 미리보기, 오류 검증, 중복 감지가 거의 모든 업로드 메뉴에서 요구됨. Apache POI 기반 공통 컴포넌트로 템플릿화.
- **알림 발송 서비스**: SMS/카카오 알림톡/FCM Push를 추상화한 `NotificationService`. 발송 채널을 메뉴별 코드에서 분기하지 않고 한 곳에서 관리.

### 3.2 외부 연동 공통 레이어

PG, 키오스크, 디멤버, 방화벽 등 5개 이상의 외부 시스템이 연동되므로, 각각 따로 클라이언트를 만들면 장애가 전파되기 쉽다.

- 외부 호출용 공통 HTTP 클라이언트(재시도, 타임아웃, Circuit Breaker — Resilience4j 권장)
- Webhook 수신용 공통 서명 검증 레이어 (PG 콜백, 키오스크 webhook 모두 위변조 검증 필요)
- 지점별 외부 연동 키/시크릿을 분리 저장하는 설정 테이블 (방화벽, PG 가맹점 코드 등 지점마다 다를 수 있음)

### 3.3 연도/지점 데이터 버전 관리

"전년도 복사", "타지점 가져오기"가 학과/전형/반/커리큘럼/상벌점/교습비/시간표/독서실 배치 등 거의 모든 기초 데이터 메뉴에서 요구된다. 단순 row copy로는 안 되고, 엔티티 간 의존관계(반 → 학과 → 전형 → 커리큘럼)를 따라가는 깊은 복사 로직이 필요하다.

- 모든 기초 데이터 엔티티에 `year`, `branch_id` 컬럼을 표준으로 둔다.
- "복사" API는 단순 INSERT SELECT가 아니라, 의존 엔티티 그래프를 순회하며 새로운 PK로 매핑하는 복사 서비스(`YearlySnapshotService`)로 추상화한다.
- 부분 수정(복사 후 일부만 수정)을 지원하려면 복사된 레코드와 원본 레코드 간 출처 추적(`copied_from_id`)을 남겨두는 것을 권장.

⚠️ 단, "학원 전송 관리"(관리자 메뉴)의 실제 용도가 PRD에 명확히 정의되어 있지 않음. 이 메커니즘이 타지점 복사 기능의 기반이 되는지 미팅에서 확인 필요.

## 4. 데이터 마이그레이션

대성전산 DB → Supabase 이관 시 고려사항:

- 계약 종료 시점이 미정인 상태이므로, 병행 운영 기간을 둘 경우 대성전산 DB ↔ 신규 DSA 양방향 동기화 어댑터가 필요할 수 있음 (단방향 1회 마이그레이션을 가정하지 말 것).
- 디멤버 급식신청 데이터처럼 대성전산 DB를 경유해 들어오는 데이터는, 신규 DSA가 대성전산 DB 접근 권한을 잃는 시점에 연결이 끊기므로 별도 설계가 필요함 (4.21 참고).

## 5. 결제(PG) 연동 아키텍처

PRD상 결제 흐름은 최소 두 갈래로 분리된다: ① 등록비/가상계좌(학원 명의 추정), ② 도시락 충전금(식당 벤더 명의 추정). 계약 주체가 다르면 연동도 분리 설계해야 한다.

### 5.1 어댑터 패턴

```java
interface PaymentGatewayClient {
    PaymentResult requestVirtualAccount(PaymentRequest req);
    PaymentResult requestCardPayment(PaymentRequest req);
    void verifyWebhookSignature(WebhookPayload payload);
}

class TuitionPgClient implements PaymentGatewayClient { ... }   // 등록비 (학원 명의)
class LunchVendorPgClient implements PaymentGatewayClient { ... } // 도시락 (벤더 명의)
```

비즈니스 로직은 어떤 PG를 쓰는지 알 필요 없이 이 인터페이스만 보고 동작하도록 설계한다.

### 5.2 Webhook 분리

- `/webhooks/pg/tuition`, `/webhooks/pg/lunch` 등 PG별로 엔드포인트를 분리하고, 서명 검증 로직도 PG별로 둔다.
- 가상계좌(무통장입금)는 카드결제와 달리 비동기 흐름이다: 발급 요청 → 입금 대기 → 입금 확인(webhook) → 만료 처리. "입금대기" 상태와 가상계좌 만료 처리 로직이 필수.
- 1회용 가상계좌(거래 단위 발급) 지원 여부를 PG사에 확인할 것 — 미지원 시 입금자명 매칭 로직을 별도로 구현해야 함.

### 5.3 데이터 모델

```sql
payment_transactions (
  id, student_id, amount,
  payment_type,        -- TUITION / LUNCH
  pg_provider,          -- 어떤 PG사인지
  pg_transaction_id,
  status,               -- 공통 정규화 상태값 (PG별 상태명을 매핑)
  settled_at
)
```

각 PG의 상태값/정산 주기가 다르므로, 통합매출장(4.35)에 두 PG 데이터를 한 화면에 보여주려면 공통 status enum으로 정규화하는 매핑 레이어가 필요하다.

### 5.4 결제 흐름이 벤더로 직결되는 경우 주의점

벤더 PG로 결제가 직결되어도, 학생 충전금 잔액/사용내역은 결국 DSA에 동기화되어야 앱과 통합매출장에 반영된다. "벤더와의 PG 연동 필요"라는 한 문장으로는 이 동기화 요구사항이 묻히기 쉬우므로, 구체적으로는 **벤더 PG의 결제완료 webhook/조회 API를 수신하여 DSA 자체 DB에 동기화하는 별도 연동**이 필요하다고 명시한다.

### 5.5 PG사 자체 대시보드 이용 옵션의 트레이드오프

수납 프로그램 대신 PG사 자체 대시보드를 이용하면 개발 범위는 줄어들지만, 통합매출장(4.35)에서 도시락 충전금 데이터가 빠지는 트레이드오프가 생긴다. PRD가 목표로 하는 "구글시트 병행 없이 DSA 하나로 통합"과 상충될 수 있으므로, 이 옵션을 택할 경우 통합매출장 완전성을 어디까지 포기할지 미팅에서 합의가 필요하다.

### 5.6 권한 분리

벤더 명의 계정에서 API 키를 받을 때, 전체 권한(환불·정산 변경 가능) 키가 아니라 조회 전용(read-only) 키를 받을 수 있는지 PG사에 확인한다. 환불 권한이 벤더에게 있는 구조라면, DSA 쪽 환불대상 관리(4.35) 메뉴는 "조회·요청"까지만 가능하고 실제 처리는 벤더 위임이 되어야 한다.

## 6. 카카오 알림톡 연동 시 일정 리스크

알림톡은 발신 프로필 등록과 템플릿별 사전 심사(통상 영업일 1~3일)가 필요하다. 출결, 미납자 알림, 합격자 발표 등 템플릿 종류를 초기에 전부 리스트업해서 Phase 0~1 단계에서 미리 등록 신청을 넣어야, 개발 완료 후 심사 대기로 출시가 지연되는 상황을 막을 수 있다.

---

## 7. 보안/권한 설계

개인정보(전화번호, 주소, 생년월일) 컬럼이 다수 메뉴에 노출되어 있다(강사 주소록, 수강생 대장 등). RBAC을 메뉴별로 나중에 끼워 넣지 말고, **Phase 0 공통 백엔드 구축 단계에서 Spring Security 기반 인가 레이어를 먼저 설계**한다. PRD 로드맵상 "관리자 권한 고도화"가 Phase 4(마지막)에 있는데, 개인정보 노출 메뉴는 Phase 1~3에서 먼저 개발되므로 순서를 재조정할 필요가 있다.

## 9. 권장 Phase 0 작업 범위 (제안)

기존 PRD의 Phase 0("DB 설계, 공통 백엔드 구축, 키오스크 연동 스펙 확인")에 아래를 추가 제안한다.

- 5개 선결 조건(2장) 확인 결과 반영
- 공통 모듈 설계: 학생 검색, 엑셀 Import/Export, 알림 서비스, 외부 연동 게이트웨이(Circuit Breaker, Webhook 서명 검증)
- RBAC 기본 골격 (메뉴별 도입이 아니라 공통 레이어로)
- PG 연동 구조 설계 (어댑터 패턴, 거래 데이터 모델)
- 알림톡 템플릿 사전 등록 신청 착수
